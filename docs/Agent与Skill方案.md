# 职能沟通翻译助手：Agent + Skill 技术方案

## 一、整体思路

用 **Agent 作为编排层**，把「产品→开发」「开发→产品」两种翻译能力做成 **Skill**，由 Agent 根据用户选择（或自动识别）调用对应 Skill，并支持流式输出。Agent 与 Skill 均为本项目自有的架构设计，不依赖任何外部平台。

---

## 二、架构概览

```
┌─────────────────────────────────────────────────────────────────┐
│                        用户界面 (Web / CLI)                        │
│  选择方向 + 输入内容 → 流式展示翻译结果                              │
└───────────────────────────────┬─────────────────────────────────┘
                                │
┌───────────────────────────────▼─────────────────────────────────┐
│                    翻译 Agent（编排层）                             │
│  • 接收输入、方向（或自动识别）                                      │
│  • 选择并调用对应 Skill                                            │
│  • 流式调用 LLM，把 chunk 推回前端                                  │
└───────────────────────────────┬─────────────────────────────────┘
                                │
        ┌───────────────────────┼───────────────────────┐
        │                       │                       │
┌───────▼───────┐     ┌─────────▼─────────┐     ┌───────▼───────┐
│ Skill:        │     │ Skill:            │     │ (加分)        │
│ 产品→开发     │     │ 开发→产品         │     │ 场景识别      │
│ 翻译          │     │ 翻译              │     │ 或 第三角色   │
│               │     │                   │     │               │
│ • 角色设定    │     │ • 角色设定        │     │ • 轻量分类    │
│ • 输出结构    │     │ • 输出结构        │     │ • 路由到上述  │
│ • 补充要点    │     │ • 补充要点        │     │   Skill       │
└───────────────┘     └───────────────────┘     └───────────────┘
        │                       │
        └───────────────────────┼───────────────────────┘
                                │
┌───────────────────────────────▼─────────────────────────────────┐
│                    LLM 调用层（统一接口）                           │
│  支持 OpenAI / Claude / 国内大模型，流式返回                        │
└─────────────────────────────────────────────────────────────────┘
```

- **Agent**：只做「输入 → 选 Skill → 流式调用 → 输出」，不写死具体提示词。
- **Skill**：每个翻译方向一个 Skill，内部包含「角色设定 + 提示词 + 输出结构约定」；在项目内用代码实现为可插拔模块，Agent 按方向加载并调用。

## 三、与题目要求的对应

| 要求 | 实现方式 |
|------|----------|
| 产品→开发、开发→产品两方向 | 两个 Skill 分别实现，Agent 按方向路由 |
| 任意大模型 API | 抽象 LLM 层，可配置 API Key 与模型 |
| 提示词体现角色差异 | 每个 Skill 内独立写 system/user prompt，明确产品/开发关注点 |
| 选择方向 + 输入 + 查看结果 | 界面提供方向选择、文本框、结果区 |
| 流式输出 | Agent 调用 LLM 流式接口，SSE 或 WebSocket 推给前端 |
| README + 2 个测试用例 + 提示词说明 | 见下方「交付物」 |
| 加分：识别需求/技术方案 | Agent 内增加「场景识别」小步骤，再路由到对应 Skill |
| 加分：补充缺失信息 | 在 Skill 的 prompt 中要求模型主动列出「待澄清点」「建议补充」 |
| 加分：第三角色 | 新增「运营/管理层」Skill，Agent 支持三向路由 |

## 四、Skill 在项目中的实现

- 目录：`skills/`（Eino ADK 标准 Skill 目录）。
- 每个 Skill 是一个目录，核心是 `SKILL.md`（YAML frontmatter + 指令正文），例如：
  - `skills/translate-product-to-dev/SKILL.md`
  - `skills/translate-dev-to-product/SKILL.md`
- Agent 侧启用 **Eino Skill middleware**：
  - 启动时发现可用 Skill（读取 name/description）
  - 运行时按 direction 选择并加载对应 Skill（通过 `skill` 工具）
  - 严格按 Skill 指令输出（补全缺失信息、结构化输出）

## 五、技术选型

| 层级 | 选型 | 说明 |
|------|------|------|
| 后端 | **Go + Gin**（或 Echo） | Go 语言，Web 框架使用 Gin；若选用 Echo 亦可，二者均支持流式响应与中间件 |
| 前端 | 单页 HTML + JS | 静态页面，通过 fetch + SSE 消费流式接口，实现打字机效果 |
| LLM | 统一封装接口 | 支持 OpenAI / Claude / 国内大模型，流式调用 |
| 数据持久化 | **本地文件** | 历史记录、配置等写入本地文件（如 JSON），不依赖数据库；见下方目录结构 `data/` |
| Agent | Go 包/结构体 | 输入 → 解析方向/场景 → 选 Skill → 调 LLM 流式接口 |
| Skill | 每个方向一个包或文件，见第四节 | 可插拔、可配置（Go 接口 + 实现） |

## 七、两个核心 Skill 的提示词设计要点

### 7.1 Skill：产品→开发

- **角色**：你是「产品需求 → 技术语言」的翻译，面向开发工程师。
- **输入**：产品侧的需求描述（可能偏业务、偏目标）。
- **输出必须包含**：推荐算法/技术方案类型、数据来源与处理、性能与实时性、预估工作量；并可主动列出「待产品确认」项。
- **补充缺失**：若需求很模糊，用 bullet 列出「技术侧需要澄清的点」和「建议补充的信息」。

### 7.2 Skill：开发→产品

- **角色**：你是「技术方案 → 业务语言」的翻译，面向产品经理。
- **输入**：开发侧的技术描述（如 QPS、优化、架构变更）。
- **输出必须包含**：对用户体验的影响、对业务/增长的支持、成本或效率的商业价值；并可主动列出「建议产品侧关注」的指标或风险。
- **补充缺失**：若技术描述缺少业务含义，用 bullet 列出「可对外传达的价值点」和「建议补充的度量」。

以上两条可直接写入各 Skill 模块的 prompt 字符串或同目录下的配置文件，保持一致。

## 八、流式输出与数据持久化

### 流式输出

- **后端（Gin）**：使用 `http.ServerSentEvents` 或自定义 `StreamingResponse`，从 LLM 流式 API 读取 chunk，按 SSE 格式（`data: {...}\n\n`）或逐行 JSON 写回 `ResponseWriter`；注意设置 `Content-Type: text/event-stream` 和 `Flush()`。
- **前端**：`EventSource` 或 `fetch` 读 stream，逐段写入结果 DOM，实现「打字机」效果。
- **Agent**：选 Skill、构造消息、调用 LLM 流式接口，将读到的 chunk 通过 channel 或 callback 推给 HTTP 层。

### 数据持久化（本地文件）

- **存储内容**：翻译历史（方向、输入、输出、时间戳）、可选配置（如默认模型、API 端点）。
- **格式**：JSON 便于读写与扩展；大文件可按日期或条数分片（如 `history_2025-03.json`）。
- **位置**：`data/` 目录，程序启动时若不存在则创建；敏感配置建议仅用环境变量，不落盘。
- **实现**：`internal/storage/file.go` 提供 `SaveHistory(record)`、`LoadHistory(limit)`、`SaveConfig`/`LoadConfig` 等，使用 `encoding/json` + `os.ReadFile`/`os.WriteFile` 或 `bufio` 追加写入。

## 九、加分项实现简述

- **场景识别**：在 Agent 内先调用一次 LLM（或小模型）做短文本分类：「需求讨论」 vs 「技术方案」，再根据结果选 product_to_dev 或 dev_to_product；若用户已选方向则跳过。
- **补充缺失信息**：已在 Skill 的 prompt 中约定，无需额外模块。
- **第三角色**：新增 `dev_to_ops.go` 或 `ops_to_product.go` 等，在 Agent 的 Skill 注册表中增加方向与路由即可。

## 十、交付物自检

- **可运行代码**：`go.mod` 依赖 + 一键启动（如 `go run ./cmd/server`），环境变量配置 API Key；数据目录 `data/` 自动创建或在 README 中说明。
- **README**：快速开始（安装 Go、配置 API Key、运行）、功能说明、至少 2 个测试用例（产品→开发、开发→产品各一）、提示词设计说明（可引用本方案第七节）。
- **提示词设计说明**：在 README 中简述「产品视角 vs 开发视角」的差异、每个 Skill 的关键设计点、如何要求模型补充缺失信息（可引用本方案第七节）。

---

## 十一、小结

- **Agent**：负责输入解析、方向/场景选择、调用对应 **Skill**、流式输出，不承载具体领域知识。
- **Skill**：承载「产品→开发」「开发→产品」的提示词与输出约定，采用 **Eino ADK 标准 Skill（`skills/*/SKILL.md`）**。Agent 通过 **Eino Skill middleware** 按 direction 加载对应 Skill，并按其指令输出，实现「在 agent 中运用 skill」。
- **数据**：历史与配置等保存在本地文件（`data/`），不依赖数据库。
- 按上述结构实现即可满足题目必做与加分项，并在文档中体现 Agent 与 Skill 的分工与设计思路。
